# Rename: Seyir -> LogHarbor

Date: 2026-07-15
Status: Executed

(Note: the old name is written as "S-e-y-i-r" without dashes throughout this doc; it is
the one file that must keep mentioning the old name, so it was restored after the
repo-wide bulk replace ran over it.)

## Context

The project name "Seyir" does not communicate the product's purpose to a non-Turkish
audience. The owner wants an English/international name where the purpose ("this is a
log tool") is evident from the name itself, ideally with a log/record connotation.

Chosen name: **LogHarbor**. "Log" up front makes the purpose instantly readable;
"harbor" keeps a quiet nod to the maritime origin of "Seyir" (logs sail in, moor, and
become searchable). Web search on 2026-07-15 found no existing project, package, or
product named LogHarbor. Rejected finalists: Logbook (collides with Zalando's
well-known Java logging library), LogVault / LogPort / LogDeck / LogBay (all taken by
existing logging products), LogHaven / LogPier / LogCove / LogBeacon (free, but weaker
fit).

## Naming decisions

| Old | New |
|---|---|
| Seyir (brand, UI title) | LogHarbor |
| Seyir.Api / Seyir.Core / Seyir.Tests / Seyir.sln | `LogHarbor.Api` / `LogHarbor.Core` / `LogHarbor.Tests` / `LogHarbor.sln` |
| SeyirDb, SeyirApiFactory classes | `LogHarborDb`, `LogHarborApiFactory` |
| X-Seyir-ApiKey header | `X-LogHarbor-ApiKey` (`X-Seq-ApiKey` alias unchanged) |
| Seyir__DatabasePath / Seyir:DatabasePath config | `LogHarbor__DatabasePath` / `LogHarbor:DatabasePath` |
| data/seyir.db default DB path | `data/logharbor.db` |
| seyir_ API token prefix | `logharbor_` |
| seyir_session cookie | `logharbor_session` |
| --seyir-* CSS custom properties | `--logharbor-*` |
| Docker image/container seyir, volume seyir-data | `logharbor`, `logharbor-data` |
| Frontend title, package name | LogHarbor / `logharbor` |

No backward-compatibility aliases for the old names: the project is pre-release and
self-hosted only. Existing API keys in a dev database keep working (authentication is
hash-based); only newly generated tokens get the `logharbor_` prefix. The existing dev
database file was renamed in place, so its data is preserved.

## Approach (as executed)

Scripted, case-aware bulk replace across the whole repo:

- Old name -> new name in three case variants (PascalCase -> `LogHarbor`,
  lowercase -> `logharbor`, UPPERCASE -> `LOGHARBOR`).
- One rule, no per-file exceptions, applied to every text file except `**/bin/`,
  `**/obj/`, `node_modules/`, and `.git/`. Binary files skipped automatically.
- Stale `bin/` and `obj/` build outputs deleted instead of edited; regenerated by the
  post-rename build.
- File and directory renames: solution file, the three project dirs and `.csproj`
  files, the two class files, and the dev database file (+ its -shm/-wal sidecars).
- Drive-by fix: `docker-compose..yml` (double-dot typo) renamed to
  `docker-compose.yml`.
- Backup taken to the session scratchpad (`seyir-backup-before-rename.tgz`) before any
  change; git was deliberately not initialized (owner's call — repo not published yet).

## Verification

1. `dotnet test backend` passes (per-user SDK: set DOTNET_ROOT/PATH to `~\.dotnet`
   first).
2. `npm run build` and `npm run test` pass in `frontend/`.
3. Final sweep: case-insensitive search for the old name outside the excluded build
   dirs returns zero matches (this spec file is the single intentional exception).

## Out of scope

- Renaming the repo folder `C:\Users\shyuuhei\GIT\Seyir` -> `...\LogHarbor`: done

- Registering domains, GitHub org/repo names, or package registry names.
