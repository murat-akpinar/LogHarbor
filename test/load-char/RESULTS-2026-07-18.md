# Load Characterization — Results (2026-07-18)

**Question:** does the ingestion write path need the background-channel
refactor now? **Answer: no — not at any realistic volume for this class of
deployment.** Details below; raw data in `/root/load-char/results.jsonl` on
the test server.

## Setup

- Test server 192.168.1.131 (LogHarbor in Docker, SQLite on a named volume),
  baseline 12,815 events / 5.3 MB, traffic-sim's ambient ~7 ev/min running
  throughout.
- Driver: 6 concurrent senders, 50/50 CLEF (batches of 100) / OTLP-JSON
  (batches of 100–200), all events `Source="load-char"`.
- Pre-test snapshot taken via read-only `VACUUM INTO`
  (`/root/load-char-pretest-20260718.db`).

## Numbers

| | Phase 1 — 300 ev/s, 15 min | Phase 2 — 1000 ev/s, 4.5 min |
|---|---|---|
| Achieved throughput | ~300 ev/s (150.5 + 149.9) | ~1089 ev/s (560.6 + 528.2) |
| Errors (non-2xx, timeouts) | **0** | **0** |
| Batch latency p50 | 7.3–7.4 ms (100 events) | 13.9–14.5 ms (200 events) |
| Batch latency p90 | 10.1–10.5 ms | 26.5–37.0 ms |
| Batch latency p99 | 41.7–42.1 ms | 88.1–89.5 ms |
| Batch latency max | 81 ms (CLEF) / 1.12 s (OTLP, single outlier) | 1.16 s (CLEF, single outlier) |

- Total ingested: **540,000 events in 19.1 min**, zero rejections.
- DB growth: 5.3 MB → 181.8 MB ≈ **342 bytes/event** including FTS.
- CLEF vs OTLP: statistically indistinguishable — encoding is not a factor.
- Post-load reads: 50k-row COUNT answers in **8.4 ms** on the 553k-row DB;
  container at 108 MiB RSS, idle CPU 0.3 %.
- Marginal write cost ≈ 0.07 ms/event at both operating points — the writer
  was nowhere near saturation even at 1000 ev/s (≈ 86M events/day pace).

## Interpretation

The current write path (per-request `WriteBatch`, SQLite WAL) holds a p99
under 90 ms at 1000 ev/s sustained with six concurrent senders and returns
zero errors. That is two to three orders of magnitude above the product's
realistic traffic. **The channel refactor (todo.md Phase 13) is not urgent**;
revisit only if a real deployment shows contention — the signal to watch is
ingest p99 or `SQLITE_BUSY` retries, not throughput.

## Caveats

- Senders batched 100–200 events/POST (as real sinks do). Unbatched
  1-event-per-POST at high concurrency was not tested — the per-key rate
  limit (1200/min) throttles that shape long before the writer would.
- The during-load monitor sample was lost to a CRLF bug (fixed in
  `.gitattributes`); CPU/read-latency under load were not captured. Ingest
  latencies and post-load reads bound the impact well enough for this
  decision.
- Single instance, single API key, LAN client.
